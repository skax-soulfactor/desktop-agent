# 컴퓨터 제어 (Computer Use) 설계

에이전트가 화면을 보고 마우스·키보드를 직접 조작해 GUI 애플리케이션을 제어하는 기능의 상세 설계.
**구현 보류 상태** — 효율(속도·비용) 문제와 완화 전략을 검토한 뒤 진행 여부를 결정한다.
본체 설계는 [DESIGN.md](DESIGN.md) 참조.

---

## 1. 목표와 비목표

**목표**
- GUI로만 가능한 작업의 자동화: 웹/데스크톱 앱에서 버튼 클릭, 폼 입력, 메뉴 조작, 데이터 옮겨 적기.
- 기존 워커(서브 에이전트) 구조에 도구 세트로 통합 — 작업 칩, 활동 로그, 취소, 승인 체계 재사용.

**비목표**
- 셸·파일 도구로 가능한 작업의 대체 (그쪽이 항상 더 빠르고 싸다 — 라우팅 규칙으로 강제).
- 실시간 반응이 필요한 작업(게임, 영상 편집 드래그 등).
- 무인 장시간 운행 (사용자가 지켜보는 반자동 운용을 전제).

## 2. 효율 문제 분석 — 왜 느리고 비싼가

순수 픽셀 루프(스크린샷 → 모델 → 클릭 반복)의 비용 구조:

| 항목 | 수치 (추정) |
|---|---|
| 스텝당 왕복 | 스크린샷 캡처 ~0.2s + 모델 추론 2~5s + 행동 실행 ~0.3s ≈ **3~6초/스텝** |
| 스크린샷 토큰 | 1440×900 다운스케일 기준 이미지당 ~1,100 토큰 (Anthropic 기준) |
| 히스토리 누적 | 매 스텝 이전 스크린샷들이 컨텍스트에 쌓임 → 스텝 수에 따라 비용이 **제곱**으로 증가 |
| 20스텝 작업 예 | 이미지 토큰만 20~200k 토큰, 소요 1~2분, Sonnet 기준 $0.5~1.5 |

핵심 결론: **픽셀 루프는 최후 수단으로만 쓰고, 아래 완화 전략으로 스텝 수와 이미지 토큰을 줄이는 것이
설계의 중심이 되어야 한다.**

### 2.1 완화 전략 (우선순위순)

1. **계층적 폴백 — GUI 제어는 3순위**
   - 1순위: 셸/API (`shell_exec`, `web_fetch`) — 파일·데이터 작업은 GUI 불필요.
   - 2순위: **스크립팅 인터페이스** — macOS AppleScript/JXA(`osascript`)는 많은 앱(Finder, Safari,
     Mail, Notes, Excel 등)을 좌표 없이 명령으로 제어 가능. 스크린샷 0장으로 끝나는 경우가 많다.
   - 3순위: **접근성(Accessibility) 트리 기반 제어** — 화면을 이미지가 아니라 UI 요소 트리(텍스트)로
     읽고, 요소 ID로 클릭. 이미지 토큰 → 텍스트 토큰(수백 토큰)으로 대체되고 좌표 오차가 없다.
     macOS AXUIElement, Windows UIAutomation, Linux AT-SPI.
   - 4순위: 픽셀 루프 (접근성 트리가 없는 캔버스형 UI, 게임형 UI만).
2. **오래된 스크린샷 히스토리 제거** — 컨텍스트에는 최신 1~2장만 유지하고 이전 것은
   "이전 화면: ~였음" 텍스트 요약으로 치환. 제곱 증가를 선형으로 낮춘다.
3. **다운스케일 + 영역 캡처** — 전체 화면 대신 대상 창만 캡처(`desktopCapturer` 창 단위),
   1280px 폭으로 다운스케일. 토큰 30~60% 절감.
4. **행동 배칭** — 모델이 한 번에 행동 시퀀스(클릭→타이핑→Enter)를 계획하고, 화면이 예상과
   다를 때만 재캡처. 확실한 구간에서 왕복을 1/3로 줄인다.
5. **매크로 기록·재생** — 성공한 작업의 행동 시퀀스를 지식베이스에 저장(`reference` 타입),
   같은 요청 반복 시 모델 호출 없이 재생하고 실패 시에만 루프로 폴백. 반복 업무의 비용을 0에 수렴시킨다.
6. **등급 라우팅** — 컴퓨터 제어 작업은 좌표 정확도가 필요하므로 `advanced` 등급 강제
   (computer-use 훈련된 모델: Claude Sonnet 계열 권장).

## 3. 아키텍처

```
메인 에이전트
  └─ delegate_task(tier=advanced, computerUse=true)
       └─ 워커 서브 에이전트 (컴퓨터 제어 도구 세트 활성화)
            ├─ [1순위] shell_exec / osascript
            ├─ [2순위] ax_read_tree / ax_click (접근성)
            └─ [3순위] screen_capture / mouse / keyboard (픽셀 루프)
       모든 도구 호출 → Permission Gateway → 활동 로그
```

- 컴퓨터 제어는 **워커 전용** 도구 세트. 메인 에이전트는 위임만 한다.
- `delegate_task`에 `computerUse: boolean` 파라미터 추가 — true일 때만 제어 도구가 워커에 노출되고,
  시작 전 "컴퓨터 제어 세션" 승인을 받는다 (4장).
- 제어 세션 동안 워커의 시스템 프롬프트에 계층적 폴백 규칙(2.1-1)을 명시해 픽셀 루프 남용을 막는다.

## 4. 승인·안전장치

도구 호출 단위 승인(클릭마다 다이얼로그)은 사용 불가능하므로 승인 단위를 세션으로 올린다.

| 장치 | 내용 |
|---|---|
| 세션 시작 승인 | "에이전트가 화면을 보고 마우스·키보드를 제어합니다" 다이얼로그 1회. 대상 앱/작업 요약 표시 |
| 상시 정지 | 작업 칩의 취소 버튼 + 전역 단축키(예: ⌘⇧.) + **마우스를 화면 좌상단 모서리로 이동 시 즉시 중단** |
| 행동 로그 | 모든 클릭·입력·캡처를 활동 로그에 실시간 기록 (기존 WorkLog 재사용). 입력 텍스트는 마스킹 옵션 |
| 위험 화면 가드 | 결제·비밀번호·시스템 설정·삭제 확인 다이얼로그 감지 시 일시정지 후 재승인 요청 (모델 판단 + 창 제목 휴리스틱) |
| 민감정보 차단 | 비밀번호 입력은 도구 차원에서 거부 (`keyboard_type`에 secure field 감지 시) |
| 스텝·시간 상한 | 세션당 최대 50스텝 / 10분. 초과 시 자동 중단 후 보고 |

감사 로그(audit.jsonl)에는 세션 시작/종료와 스텝 수를 기록한다.

## 5. 도구 명세

### 5.1 스크립팅·접근성 (권장 경로)

```typescript
// macOS 우선. Windows/Linux는 8장.
osascript_run     { script: string }                       // AppleScript/JXA 실행. risk: execute
ax_read_tree      { app?: string; windowTitle?: string }   // 전면 창의 UI 요소 트리(텍스트) 반환. risk: read
                  // → [{ id, role, title, value, enabled, frame }] 최대 300개, 깊이 8
ax_click          { elementId: string }                    // 트리의 요소 id로 클릭 (좌표 불필요). risk: execute
ax_set_value      { elementId: string; value: string }     // 텍스트 필드 입력. risk: execute
```

### 5.2 픽셀 루프 (폴백 경로)

```typescript
screen_capture    { windowTitle?: string; region?: {x,y,w,h} }  // 다운스케일 PNG + 스케일 팩터 반환. risk: read
mouse_action      { action: 'click'|'double'|'right'|'move'|'drag'; x: number; y: number;
                    toX?: number; toY?: number }                // 논리 좌표 (스케일 보정은 도구 내부에서). risk: execute
keyboard_type     { text: string }                              // risk: execute
keyboard_key      { combo: string }                             // 예: "cmd+s", "enter". risk: execute
scroll            { x: number; y: number; deltaY: number }      // risk: execute
```

### 5.3 좌표계 규칙

- 모델에는 항상 **다운스케일된 이미지의 좌표계**로 묻고, `screen_capture`가 반환한
  `scaleFactor`(물리/논리 비율 × 다운스케일 비율)로 도구 내부에서 실좌표 변환한다.
  모델이 스케일을 알 필요가 없게 하는 것이 오차를 줄이는 핵심.
- 멀티 모니터: v1은 주 모니터만. 창 단위 캡처 시 창의 화면 원점 오프셋을 함께 보정.

## 6. 워커 루프 확장

기존 `runTask` 루프를 그대로 쓰되, 제어 세션에서 두 가지가 추가된다.

1. **스크린샷 히스토리 프루닝**: 도구 결과 중 이미지가 포함된 메시지는 최신 2개만 유지,
   이전 것은 `[이전 화면 캡처 — 생략됨]` 텍스트로 교체 후 재호출. (2.1-2)
2. **행동 계획 강제**: 시스템 프롬프트에 "행동 전 반드시 ax_read_tree를 먼저 시도하고,
   픽셀 캡처는 접근성 트리에 대상이 없을 때만" + "확실한 시퀀스는 한 번에 계획하고
   화면 검증은 시퀀스 끝에서 1회" 규칙 명시. (2.1-1, 2.1-4)

매크로 기록(2.1-5)은 v2: 성공 세션의 행동 시퀀스를
`{ trigger: 요청 패턴, steps: [...도구 호출들] }` 형태로 지식베이스에 저장하고,
유사 요청 시 워커가 먼저 재생을 시도한다.

## 7. OS 권한 온보딩

| OS | 필요 권한 | 안내 방식 |
|---|---|---|
| macOS | 화면 기록(캡처), 손쉬운 사용(입력·AX), 자동화(osascript 대상 앱별) | 최초 사용 시 설정 딥링크(`x-apple.systempreferences:`)와 함께 단계 안내 다이얼로그 |
| Windows | 별도 권한 없음 (관리자 권한 창 제어는 불가 — 명시적 거부) | — |
| Linux | X11: xdotool 설치 안내. Wayland: xdg-desktop-portal 제한적 지원 (v1 제외 권장) | 배포판별 설치 명령 안내 |

## 8. OS별 구현 매핑 (네이티브 모듈 없이)

| 기능 | macOS (v1) | Windows (v2) | Linux X11 (v2) |
|---|---|---|---|
| 캡처 | Electron `desktopCapturer` | 동일 | 동일 |
| 마우스/키보드 | `cliclick`(brew) 또는 `osascript` System Events | PowerShell + user32/SendKeys | `xdotool` |
| 접근성 트리 | `osascript` System Events UI elements (또는 Swift 헬퍼 바이너리) | UIAutomation (PowerShell) | AT-SPI (`pyatspi`/CLI) |
| 앱 스크립팅 | AppleScript/JXA | COM (Office 등) | D-Bus (일부 앱) |

nut.js 유료화로 라이브러리 대신 OS 유틸리티 방식을 채택한다 (프로젝트의 무-네이티브모듈 방침 유지).
접근성 트리 품질이 osascript로 부족하면 v2에서 작은 Swift/C# 헬퍼 바이너리를 동봉하는 것을 검토.

## 9. 비용·성능 추정 (전략 적용 후)

시나리오: "메일 앱에서 특정 메일 찾아 첨부파일 저장" (~15 논리 스텝)

| 방식 | 스텝 수 | 이미지 전송 | 예상 비용(Sonnet) | 예상 시간 |
|---|---|---|---|---|
| 순수 픽셀 루프 | 15 | 15장 (히스토리 누적) | ~$1.0 | ~90s |
| + 히스토리 프루닝·다운스케일 | 15 | 15장 (컨텍스트엔 2장) | ~$0.25 | ~75s |
| 접근성 트리 우선 | 4~6 | 0~2장 | ~$0.05 | ~25s |
| AppleScript 직접 제어 | 1~2 | 0장 | ~$0.01 | ~8s |

**결론: 효율 문제의 답은 "픽셀 루프를 잘 만드는 것"이 아니라 "픽셀 루프를 최대한 안 쓰는 것"이다.**
접근성/스크립팅 경로가 커버하는 작업 범위가 충분히 넓다면 투자 가치가 있고,
픽셀 루프만 필요한 작업이 대부분이라면 보류가 맞다.

## 10. 단계별 로드맵과 go/no-go 기준

| 단계 | 범위 | 판단 기준 |
|---|---|---|
| **CU-0 (검증)** | osascript + ax_read_tree 프로토타입으로 대표 작업 3개(메일 첨부 저장, 웹 폼 입력, 앱 간 복사) 수동 측정 | 접근성 경로로 2/3 이상 성공 + 작업당 <$0.1 이면 진행 |
| CU-1 | macOS: 5.1 도구 + 세션 승인 + 정지 + 행동 로그 | — |
| CU-2 | 픽셀 루프 폴백 (5.2) + 히스토리 프루닝 + 위험 화면 가드 | — |
| CU-3 | 매크로 기록·재생, Windows 지원 | — |

CU-0는 하루 규모의 실험으로, 본 구현 없이 효율 우려를 데이터로 검증할 수 있다.
